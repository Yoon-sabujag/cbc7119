# Quick 260424-7l3: CheckpointsPage 마커 편집 라우팅 버그

## One-liner
CheckpointsPage 가 유도등 마커 (id 프리픽스 `FPM-`) 편집 시에도 `PUT /api/check-points/:id` 를 호출해 404 를 내던 버그를 수정. id 로 분기하여 마커는 `floorPlanMarkerApi.update` 로 라우팅.

## 배경
DevTools 콘솔 리포트:
```
PUT /api/check-points/FPM-MwBdV9omhcbN 404 (Not Found)
```
CheckpointsPage 는 `check_points` 와 `floor_plan_markers` (유도등) 를 동일한 `CheckPointFull` 형태로 섞어 리스트한다. 그러나 `updateMutation` 은 id 와 무관하게 항상 `checkPointApi.update` 를 호출했음. 마커 id (`FPM-...`) 에 대해서는 해당 endpoint 에 리소스가 없어 404 가 발생.

## 구현 요약

`src/pages/CheckpointsPage.tsx` — `CheckPointModalContent` 컴포넌트 내 mutation 로직 수정.

```ts
const isMarker = !!cp?.id?.startsWith('FPM-')

const updateMutation = useMutation({
  mutationFn: async (data): Promise<void> => {
    if (isMarker) {
      await floorPlanMarkerApi.update(cp!.id, {
        label: data.location,
        description: data.description ?? null,
        zone: (data.zone as string) ?? null,
      })
      return
    }
    await checkPointApi.update(cp!.id, data)
  },
  ...
})
```

- 마커: label (=개소명), description, zone 만 전송. floor/locationNo/category/isActive 는 마커 스키마에 없거나 의미가 달라 전달하지 않음.
- 일반 개소: 기존 동작 유지.
- `deactivateMutation` 도 동일 분기. 마커는 `isActive` 컬럼이 없어 대신 `floorPlanMarkerApi.delete` 로 fallback (안전 조치 — UI 는 마커 모달에서 해당 버튼을 숨기는 게 더 낫지만 현재 범위 밖).
- React Query 무효화: `['floorplan-markers-all']` 도 함께 무효화 → 리스트 즉시 갱신.
- `mutationFn` 리턴 타입을 `Promise<void>` 로 명시 → 타입 유니온 (Promise<void> | Promise<CheckPointFull>) 컴파일 에러 해결.

## Task 2 — Bug J fix (FloorPlanPage 점검 입력 모달 AccessBlockedPopup 반영)

### 증상

사용자 증언 (2026-04-24):
> "FloorPlanPage 에서 접근불가 마커 클릭 → '점검 기록 입력' 버튼 → FloorPlan 자체
> 점검 모달(InspectionPage 의 InspectionModal 과 별개)이 뜨는데, 여기서는
> AccessBlockedPopup 이 전혀 반영 안 됨. 사용자 기대: InspectionModal 진입은
> 되지만 바로 AccessBlockedPopup 뜨도록."

### Root Cause

`FloorPlanPage.tsx` 의 인라인 `inspectModal` 은 InspectionPage 의 InspectionModal 과
독립된 별도 컴포넌트. AccessBlockedPopup 분기가 포팅되지 않아 `selected.description`
에 '접근불가' 가 있어도 일반 점검 폼이 그대로 렌더됨.

`FloorPlanMarker.description` 은 `utils/api.ts` 타입에도 정의돼 있고 (260424-7l2 에서 추가),
GET 응답에도 포함되므로 (SELECT *), 클라이언트에서 이미 사용 가능한 데이터임.

### 구현

1. **Import 추가**: `import { AccessBlockedPopup } from '../components/AccessBlockedPopup'`.
2. **isAccessBlocked 플래그**: 모달 IIFE 안에서 `const isAccessBlocked = !!selected.description?.includes('접근불가')`.
   (문자열 접근 방식은 기존 InspectionPage 와 동일. '[접근불가]' 대괄호 유무 무관 — `includes` 로 부분 일치.)
3. **컨테이너 position:relative**: 기존 모달 콘텐츠 `<div>` 에 `position:'relative'` 추가.
   AccessBlockedPopup 은 `position:absolute; inset:0` 이라 부모 박스가 relative 여야 함.
4. **팝업 렌더**: 콘텐츠 `<div>` 안 최상단에 `{isAccessBlocked && <AccessBlockedPopup onConfirm={() => { setInspectModal(false); setSelected(null) }} />}`.
   FloorPlan UX 는 단일 마커 기반 (순차 피커 없음) → 확인 버튼 = 모달 닫고 마커 선택 해제.
5. **저장 버튼 이중 방어**: `disabled` 조건에 `isAccessBlocked` 추가, 라벨 '저장' → '접근 불가 개소'.
   팝업이 오버레이로 입력을 막지만 데이터 무결성 확보를 위해 추가 방어.

### 구현 비교 (InspectionPage vs FloorPlanPage)

| 항목 | InspectionPage InspectionModal | FloorPlanPage inspectModal (Bug J 후) |
| ---- | ------------------------------ | --------------------------------------- |
| 데이터 소스 | pickerSourceCPs (마커/cp 혼합) | selected (단일 FloorPlanMarker) |
| description 필드 | cp.description | selected.description |
| 팝업 우선순위 | AccessBlockedPopup > RevisitPopup | AccessBlockedPopup 만 (기존 RevisitPopup 은 점검 기록 입력 버튼 클릭 시 별도 처리) |
| 확인 버튼 동작 | advanceToNextPending(skipId, true) — 6단계 폴백 | setInspectModal(false); setSelected(null) — 단일 마커 UX |
| 저장 버튼 disabled | isAccessBlocked 포함 | isAccessBlocked 포함 |
| 라벨 변경 | '접근 불가 개소' | '접근 불가 개소' |

### 회귀 리스크

- **비-접근불가 마커**: `isAccessBlocked=false` → AccessBlockedPopup 렌더 안 됨,
  `position:'relative'` 는 레이아웃에 무관 (팝업이 absolute 로 띄워질 때만 의미),
  저장 버튼 OR 조건 추가는 `isAccessBlocked=false` 면 기존과 동일. 완전 무영향.
- **접근불가 마커 QR 스캔**: FloorPlanPage 의 인라인 점검 모달은 QR 스캔이 아닌
  마커 클릭 경로로만 열림 → QR 스캔은 InspectionPage 로 라우팅되므로 Bug J 는
  마커 직접 점검 경로에만 해당.
- **확인 버튼 = 모달 닫기**: 사용자가 다시 접근불가 마커를 클릭하면 팝업 재노출
  (매번 fresh state) — '다음 마커 이동' 이 아니라는 점이 InspectionModal 과 다르지만
  FloorPlanPage UX 는 사용자가 도면에서 마커를 직접 선택하는 구조라 자연스러움.
- **description='기타 특이사항' 케이스**: '접근불가' 키워드가 없는 일반 설명은
  `includes('접근불가')=false` → 기존 동작 유지.
