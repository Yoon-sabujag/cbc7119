# Quick 260424-7l2: 유도등 마커 description 지원

## One-liner
floor_plan_markers 에 description 컬럼을 추가하여 유도등 마커에도 check_points 와 동일한 '[접근불가]' 메모 기반 판정 + AccessBlockedPopup 연동이 가능해짐.

## 배경
유도등은 check_points 테이블이 아닌 floor_plan_markers 테이블로만 관리되어 description 컬럼이 없었음. 따라서 CheckpointsPage 에서 "[접근불가]" 같은 메모를 저장할 수 없었고, InspectionModal 의 AccessBlockedPopup 이 유도등 마커에서는 트리거되지 않았음.

## 구현 요약

### 1. DB 마이그레이션 (0072)
```sql
ALTER TABLE floor_plan_markers ADD COLUMN description TEXT;
```
- 기존 row 는 NULL
- check_points 와 대칭 구조

### 2. API 확장
- `GET /api/floorplan-markers` — `SELECT *` 가 description 을 자동 포함 (mRows → data → merged 스프레드)
- `PUT /api/floorplan-markers/:id` — body 에 `description?: string | null` 추가, 조건부 SET 절

### 3. 타입
- `FloorPlanMarker.description?: string | null` 추가
- `floorPlanMarkerApi.update` 페이로드에 description 추가

### 4. InspectionModal 매핑 (src/pages/InspectionPage.tsx `pickerSourceCPs`)
- 마커의 실제 description 이 있고 trim 결과가 비어 있지 않으면 우선 사용
- 없으면 기존 marker_type 기반 한글 라벨 (`GL_COL_LABEL`) 을 기본값으로
- 결과: 관리자가 CheckpointsPage 에서 "[접근불가]" 저장 → 점검 중 해당 마커 클릭 → AccessBlockedPopup 뜸 (기존 접근불가 로직 공용)

## Task 2 — Bug I fix (유도등 AccessBlockedPopup 확인 버튼 미동작)

### 증상

사용자 증언 (2026-04-24):
> "관리자 UI 에서 유도등 마커 description 에 [접근불가] 저장 → 정상 반영.
> 점검 모달에서 해당 마커 선택 → AccessBlockedPopup 정상 표시.
> 그런데 확인 버튼을 눌러도 반응이 전혀 없음. 일반 카테고리(소화기/완강기)는 정상."

### Root Cause

`InspectionPage.tsx` 의 `advanceToNextPending` 맨 앞 줄 `if (isGuideLight) return`
가드가 `fromAccessBlocked=true` 경로도 함께 차단. 유도등 마커의 AccessBlockedPopup
`onConfirm` 에서 `advanceToNextPending(selectedCP.id, true)` 를 호출해도 즉시 return
되어 state 변화가 전혀 없음 → `isAccessBlocked` / `showAccessBlockedPopup` 그대로 true
유지 → 팝업 그대로 렌더. 버튼이 "눌리지 않는" 것처럼 체감됨.

가드의 원래 의도는 "유도등은 handleSave 후 자동 전진하지 않는다" (기존 동작 보존)
이지만, fromAccessBlocked 를 구분하지 않아 접근불가 확인까지 함께 막힌 것.

### 수정

가드 조건을 `if (isGuideLight && !fromAccessBlocked) return` 로 변경. 변경 의미:

| 경로 | isGuideLight | fromAccessBlocked | 동작 |
| ---- | ------------ | ----------------- | ---- |
| 유도등 저장 auto-advance | true | undefined | early return (기존 유지) |
| 유도등 접근불가 확인 | true | true | 6단계 폴백 체인 진입 → 정상 동작 |
| 일반 저장 auto-advance | false | undefined | 기존 4-tier |
| 일반 접근불가 확인 | false | true | 기존 6단계 폴백 |

유도등 컨텍스트에서도 `pickerIdx / selectedFloor / selectedZone` 기반 네비게이션은
그대로 유효하므로 (pickerSourceCPs 는 zone+floor 필터 결과 마커 리스트), 6단계 폴백
체인이 그대로 작동. 잔여 접근불가 마커가 전부 dismiss 돼도 Step 6 에서
`setDismissedBlockedId(skipCpId)` 로 팝업만 닫히고 모달 유지.

- **6단계 폴백 체인 자체**: 일반 카테고리용으로 검증된 로직. 유도등에 적용해도
