# Quick 260424-7l2: 유도등 마커 description 지원

## One-liner
floor_plan_markers 에 description 컬럼을 추가하여 유도등 마커에도 check_points 와 동일한 '[접근불가]' 메모 기반 판정 + AccessBlockedPopup 연동이 가능해짐.

## 배경
유도등은 check_points 테이블이 아닌 floor_plan_markers 테이블로만 관리되어 description 컬럼이 없었음. 따라서 CheckpointsPage 에서 "[접근불가]" 같은 메모를 저장할 수 없었고, InspectionModal 의 AccessBlockedPopup 이 유도등 마커에서는 트리거되지 않았음.

## 구현 요약

### 1. DB 마이그레이션 (0072)
```sql
ALTER TABLE floor_plan_markers ADD COLUMN description TEXT;
```
- 기존 row 는 NULL
- check_points 와 대칭 구조

### 2. API 확장
- `GET /api/floorplan-markers` — `SELECT *` 가 description 을 자동 포함 (mRows → data → merged 스프레드)
- `PUT /api/floorplan-markers/:id` — body 에 `description?: string | null` 추가, 조건부 SET 절

### 3. 타입
- `FloorPlanMarker.description?: string | null` 추가
- `floorPlanMarkerApi.update` 페이로드에 description 추가

### 4. InspectionModal 매핑 (src/pages/InspectionPage.tsx `pickerSourceCPs`)
- 마커의 실제 description 이 있고 trim 결과가 비어 있지 않으면 우선 사용
- 없으면 기존 marker_type 기반 한글 라벨 (`GL_COL_LABEL`) 을 기본값으로
- 결과: 관리자가 CheckpointsPage 에서 "[접근불가]" 저장 → 점검 중 해당 마커 클릭 → AccessBlockedPopup 뜸 (기존 접근불가 로직 공용)

## 오케스트레이터 후속 작업 (필수)

**프로덕션 D1 마이그레이션 실행:**
```bash
cd cha-bio-safety
npx wrangler d1 execute cha-bio-db --remote --file=migrations/0072_floor_plan_markers_description.sql
```

**배포:**
```bash
cd cha-bio-safety
npm run deploy -- --branch production
```

**실행 순서 중요:** 마이그레이션 → 배포. 반대로 하면 배포된 코드가 아직 없는 컬럼을 SELECT * 로 읽더라도 description 필드가 응답에 없을 뿐 에러는 아니지만, PUT 시 description 을 보내면 SQL 에러가 남. 안전을 위해 마이그레이션 먼저.

## 검증 방법 (배포 후)

1. 관리자로 로그인 → 점검 개소 관리 → 카테고리 "유도등" 선택
2. 마커 하나 수정 → "설명" 필드에 `[접근불가]` 저장
3. 점검 → 유도등 → 해당 층/구역 진입 → 마커 선택 → AccessBlockedPopup 이 표시되는지 확인

## 회귀 리스크
- 낮음. `SELECT *` 와 조건부 SET 절 사용으로 기존 클라이언트/API 는 영향 없음.
- description 이 NULL 인 기존 마커는 기존 동작 (marker_type 라벨 표시) 유지.
- 마이그레이션 실행 전 배포하는 경우 PUT 에서만 에러 가능성 — 오케스트레이터가 순서 준수.
