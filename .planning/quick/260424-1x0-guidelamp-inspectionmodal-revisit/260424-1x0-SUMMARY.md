# Quick 260424-1x0: 유도등 InspectionModal 재진입 팝업 지원

## 배경

260423-htx Task 4/5 에서 "구조적 제약" 으로 미지원 표시했던 부분. InspectionModal
유도등 경로는 `useInspectionRevisitPopup` 호출부에서 `checkpointId: isGuideLight ? null : ...`
로 훅을 강제로 비활성화하고 있었다. 이유: 유도등 개소는 CheckPoint 기반이 아니라
FloorPlan 마커 기반이며 (`cp.id = 'MARKER:{markerId}'`), `monthRecords` 는 cpId 키만
쓰고 유도등은 `markerRecords` (CheckResult 만) 에 저장했기 때문에 훅이 조회할
엔트리가 없었음.

FloorPlan 마커 팝업은 이미 동작 중이며 `selected.last_result / last_inspected_at /
last_record_id / last_status / last_inspected_by` 필드를 그대로 사용 → 데이터 자체는
서버에서 오고 있었다. InspectionModal 쪽은 그 데이터를 미쓰고 있었던 것.

## 구현 접근 (옵션 C 채택)

**세 가지 옵션 중 (C) "호출부에서 monthRecords 에 마커 기반 레코드를 병합해 전달" 채택.**

- (A) 훅에 `markerRecords` 파라미터 추가 — 훅 시그니처 변경, 다른 모달에도 전파.
- (B) InspectionModal 유도등 전용 inline 체크 — FloorPlan 팝업 로직 중복.
- **(C) monthRecords 병합** — 훅 시그니처 손 안 댐. InspectionPage 1개 파일 + 공통
  컴포넌트 재사용. FloorPlan 팝업과 정확히 동일한 UX 문구/동작.

## Task 2 — Bug C fix (후속, 260424 검증)

### 증상

검증 중 "도면 점검에서 미점검으로 확인된 개소(유도등 마커) 인데 InspectionModal
유도등 진입 시 (가) completed 팝업이 떠 있음." 두 UI 의 데이터 소스가 달라
발생한 불일치.

### Root Cause

`loadTodayRecords` 의 monthData 루프가 `r.result` 유효성 검사 없이 monthMap 에
entry 를 upsert 하고 있었다. 이후 훅의 `if (!meta.result) return null` 가드가
있긴 하지만, 동일한 `monthRecords` 를 소비하는 다른 경로(doneCount, UI 판정
등)가 entry 존재만으로 "기록 있음" 을 추론할 때 오탐을 유발한다. FloorPlan
API 는 `check_records.checked_at` + result 로 이미 한 번 걸러져 `last_result`
= null 로 내려오는 반면, getMonthRecords 응답에는 result 가 빈/falsy 인
레코드도 섞여 들어올 수 있다(세션 dangling row 등 데이터 주변 엣지).

### 수정

```ts
// loadTodayRecords() 내 monthData 루프
const rawResult = (r as any).result
if (!rawResult) continue   // ← result 유효할 때만 upsert
```

cpId 기준/MARKER: 기준 양쪽 키 모두 유효 레코드에만 적재됨.

### 회귀 리스크

- 정상 경로(result 가 normal/caution/bad/unresolved/missing 중 하나인 레코드)
  에는 영향 없음. 빈 result 엔트리가 monthRecords 에 들어가던 극단 케이스만
  차단. doneCount 집계가 약간 감소하는 효과는 실제 점검 기록이 아닌 것을
  "완료" 로 세던 버그가 교정되는 방향이므로 의도된 변화.
