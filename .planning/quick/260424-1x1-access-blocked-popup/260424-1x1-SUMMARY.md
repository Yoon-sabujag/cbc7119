# Quick 260424-1x1: 접근불가 개소 자동 스킵 대신 팝업 노출

## 배경

사용자 요청:
> "접근 불가 개소를 자동 스킵하지 말고, 개소명 카드 아래로 '접근 불가 개소입니다'라는
> 팝업을 띄우고 확인이나 이전/이후 개소로 이동 버튼을 누르거나 스와이프하면
> 스킵이 되게 해야 할 것 같음."

## 적용 카테고리 (DB 조사 결과)

`cp.description` 패턴별로 현재 접근불가 CP 분포:

| 패턴                       | 카테고리 | CP 개수 | defaultResult | 기존 처리                     |
| -------------------------- | -------- | ------- | ------------- | ----------------------------- |
| `description = '접근불가'` | 완강기   | 9개     | 없음          | 모달 open 시 자동 normal 제출 |
| `description ~ '[접근불가]'` | 유도등   | 2개     | 'normal'      | defaultResult 뱃지 + 피커 제외 |

신규 팝업 UX 는 주로 **완강기 접근불가 9개** 에 적용된다. 유도등 `[접근불가]`
은 이미 `cp.defaultResult` 분기로 처리 중이며 pendingCPs 필터에서 `defaultResult`
제외 조건이 그대로 남아 있어 거기에는 변화 없음.

## 엣지 케이스

1. **층/구역 전체가 접근불가인 경우**: `firstPending` 은 -1 반환 → 인덱스 0 으로
   폴백 → 인덱스 0 이 접근불가 cp 라면 즉시 팝업 노출. 사용자는 ‹/›/스와이프/닫기
   로 탐색 가능. 수용 가능한 UX.
2. **모든 잔여 cp 가 접근불가인 상태에서 '확인' 클릭**: `advanceToNextPending` 이
   이동 대상을 못 찾고 no-op → 현재 cp 유지 → 팝업 계속 표시. 사용자가 수동으로
   "닫기" 버튼(모달 헤더 하단) 또는 이전 화살표로 빠져나감.
3. **유도등 (isGuideLight)**: `advanceToNextPending` 은 가드로 early return
   (기존 auto-advance 동작과 일치). 유도등 개소는 [접근불가] + defaultResult 조합
   이므로 `pendingCPs` 에서 이미 제외됨 → 신규 UX 영향 없음.
4. **완강기 auto-submit useEffect 타이밍**: useEffect 가 비동기라 모달 open 직후
   0~수백 ms 동안은 records 가 비어있을 수 있음. 이 타이밍에 사용자가 접근불가
   cp 를 탐색하면 팝업은 뜨지만 records 는 아직 미반영. useEffect 완료 후
   `loadTodayRecords` 가 돌아 records 가 채워짐. UX 영향 없음 (팝업의 본 기능은
   "안내" 이므로 기록 여부와 무관).
5. **QR 스캔 경로**: `initialCpId` 로 바로 특정 cp 에 포커스 가는 경로는 변경 없음.
   해당 cp 가 접근불가 cp 면 스캔 직후 팝업 노출 — 사용자가 QR 스캔 했는데 접근
   불가였음을 즉시 알 수 있어 정보 제공 측면에서 오히려 개선.

## Task 2 — Bug E fix (CRITICAL, 260424 검증)

### 증상

사용자 증언: "접근불가 팝업의 확인 버튼이 눌리지 않아 팝업이 사라지지 않음.
확인 불가."

### Root Cause

`advanceToNextPending` 의 isIncomplete 필터가 접근불가 cp 를 제외하므로,
잔여 cp 가 전부 접근불가이거나 이미 완료된 상황에서 AccessBlockedPopup
'확인' 버튼을 누르면 `nextIdx/firstIdx/nextFloor/nextZone` 모두 -1/undefined
를 반환해 **no-op** 으로 종료된다. `selectedCP` 가 그대로라 `isAccessBlocked`
역시 true 유지 → 팝업 그대로 렌더. 사용자에게는 "버튼이 먹히지 않는" 것으로
체감됨.

버튼 자체의 onClick 은 정상 바인딩되어 있고 click 이벤트도 정상 발화한다.
핸들러가 실행되긴 하지만 state 를 전혀 바꾸지 않는 것이 직접적 원인.

### 수정

`advanceToNextPending(skipCpId?, fromAccessBlocked?)` 의 두 번째 파라미터 추가.
기존 4-tier 탐색(현재 층 next → 현재 층 first → 다른 층 → 다른 구역)이 모두
실패했을 때, `fromAccessBlocked=true` 이면 추가 폴백:

1. `pickerIdx + 1 < pendingCPs.length` → setPickerIdx(pickerIdx + 1)
2. 그 외 `pickerIdx > 0` → setPickerIdx(0)  (wrap-around 한 칸 이동)
3. pendingCPs.length ≤ 1 → `onClose()` (모달 종료)

AccessBlockedPopup 호출부: `advanceToNextPending(selectedCP.id, true)`.
기존 handleSave 경로(`setTimeout(() => advanceToNextPending(justSavedId), 600)`)
는 `fromAccessBlocked` 미전달 → 원래 동작 그대로.

추가: `nextZone` 분기에 누락됐던 `return` 추가 (기존에도 폴백 도달 불가능
했지만, 이제 그 뒤에 Bug E 폴백 블록이 생겼으므로 명시 필요).

- `cha-bio-safety/src/pages/InspectionPage.tsx` (+18 / -3)

### 회귀 리스크

- 저장 후 auto-advance 경로 (`handleSave` → `setTimeout` → `advanceToNextPending(justSavedId)`)
  는 `fromAccessBlocked` 를 전달하지 않으므로 동작 불변.
- 일반 케이스(accessible next cp 존재) 는 기존 4-tier 에서 `return` 으로 조기
  종료되므로 새 폴백 블록 도달 불가능.
- onClose 로 모달 종료되는 극단 케이스는 "접근불가 단일 cp 만 남은" 상황
  뿐이라 실무상 드물지만, 사용자가 확인을 눌러도 아무 반응 없는 기존 UX

## Task 3 — Bug F fix (UX, 260424 사용자 피드백)

### 증상 / 불만

사용자 증언:
> "잔여 개소가 전부 접근불가/완료인 상황에서 접근불가 팝업의 '확인' 버튼을
> 누르면 점검 모달 자체가 닫혀버림. 다음 층에 점검할 개소가 있을 수도
> 있는데 모달이 닫히는건 불편함."

Task 2(Bug E) 수정에서 최종 폴백으로 `onClose()` 를 넣어 "확인이 반응했음" 을
보여준 것이 오히려 UX 퇴보로 체감됨. 사용자는 모달이 유지되어야 다음 층/다른
구역으로 수동 탐색이 가능함.

### Root Cause

`advanceToNextPending(skipCpId, fromAccessBlocked=true)` 의 Bug E 폴백 블록
끝 줄 `onClose()` (accessible 대상이 전혀 없고 피커 길이 ≤ 1 인 경우)가
모달을 종료시킴.

### 수정

`advanceToNextPending` 의 `fromAccessBlocked` 폴백 블록 완전 교체:

**기존 (Bug E 해결책)**:
1. `pickerIdx + 1 < pendingCPs.length` → `setPickerIdx(pickerIdx + 1)`
2. `pickerIdx > 0` → `setPickerIdx(0)`  (wrap-around)
3. 그 외 → `onClose()`

**수정 (Bug F)**:
1. `selectedFloor && selectedZone` 하에서 `availableFloors.indexOf(selectedFloor) + 1`
   위치에 다음 층이 있으면 → `setSelectedFloor(nextFloorAny); setPickerIdx(0)`
   (잔여 여부 무관. 1회만 이동. 상위 4-tier 에서 이미 '잔여 있는 층' 탐색을
   시도했으므로 여기 도달은 "잔여는 없지만 다음 층 자체는 존재" 케이스).
2. 다음 층이 없거나(현재 층이 zone 마지막 층) selectedFloor/Zone 미확정이면
   → `setDismissedBlockedId(skipCpId)` — 해당 cp 의 접근불가 팝업만 닫고
   모달은 그대로 유지. 사용자가 ‹ 화살표 / 구역·층 탭 / 닫기 버튼 등으로
   수동 탐색.

`dismissedBlockedId` state 신설 + `useEffect` 로 `currentSelCP.id` 가
dismiss 된 id 와 달라지면 자동 null 리셋 → 다른 접근불가 cp 재진입 시 팝업
정상 재노출. 같은 cp 재방문 시도에도 dismiss 는 1회성으로 자연 해제됨
(pickerIdx 이동 → 다른 cp → dismissedBlockedId null → 돌아옴 → 다시 노출).

렌더 조건: `isAccessBlocked ? ...` → `showAccessBlockedPopup ? ...`
(`showAccessBlockedPopup = isAccessBlocked && dismissedBlockedId !== currentSelCP?.id`).
저장 버튼 disabled 조건과 '접근 불가 개소' 라벨은 `isAccessBlocked` 기준을
그대로 유지 — 데이터상 접근불가인 cp 는 dismiss 여부와 무관하게 저장 불가여야
하므로 이중 방어.

### Root Cause

`advanceToNextPending` 의 `fromAccessBlocked` 폴백 체인이 불완전:
- 상위 4-tier (nextIdx / firstIdx / nextFloor with pending / nextZone with pending) 는
  `isIncomplete` 필터로 접근불가 cp 를 제외 — 같은 층 다른 접근불가 cp 를 건너뜀.
- fromAccessBlocked 폴백 블록은 `availableFloors[currIdx+1]` 만 시도 → 마지막 층 시
  즉시 `setDismissedBlockedId` 로 종료. 다음 zone 으로 진입하지 않음.

### 수정 — 폴백 체인 완전 재정의

`fromAccessBlocked=true` 경로를 **7단계 선형 체인** 으로 재구성:

| 단계 | 설명                                                 | 구현                                                                                   |
| ---- | ---------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 1    | 현재 층 accessible next cp (i > pickerIdx)           | `pendingCPs.findIndex((cp,i) => i > pickerIdx && isIncomplete(cp, skipCpId))`          |
| 2    | 현재 층 accessible first cp                          | `pendingCPs.findIndex(cp => isIncomplete(cp, skipCpId))`                               |
| 3    | 현재 층 다른 접근불가 cp (현재 제외, next)           | `pendingCPs.findIndex((cp,i) => i > pickerIdx && isBlockedOther(cp))` ← **F-a 해결**   |
| 4    | 현재 층 다른 접근불가 cp (현재 제외, first)          | `pendingCPs.findIndex(isBlockedOther)` ← **F-a 해결**                                  |
| 5    | 같은 zone 내 다음 층 (availableFloors[currIdx+1])    | `setSelectedFloor(availableFloors[currFloorIdx+1]); setPickerIdx(0)`                   |
| 6    | 다음 zone 첫 층 (availableZones[currZoneIdx+1])      | `setSelectedZone(next); setSelectedFloor(null); setPickerIdx(0)` ← **F-b 해결**        |
| 7    | 그 외 → 팝업만 닫고 모달 유지                        | `setDismissedBlockedId(skipCpId)`                                                      |

핵심 포인트:

- `isBlockedOther(cp)` 는 `cp.id !== skipCpId && !cp.defaultResult && cp.description?.includes('접근불가')`.
  즉 "방금 확인한 cp 를 제외한 동일 층 접근불가 cps" — 사용자가 같은 층에서 접근불가
  cps 를 순차적으로 순회할 수 있게 해줌.
- 단계 5 (같은 zone 내 다음 층) 는 **accessible 여부 무관** (상위 4-tier 에서
  이미 잔여 있는 층을 시도했으므로 여기선 단순히 "다음 층" 이동).
- 단계 6 (다음 zone 첫 층) 은 **availableZones 순서 유지** (ZONE_CONFIG 는
  research → office → underground 순). `selectedFloor=null` 로 두면
  availableFloors 재계산 후 useEffect(line ~2829) 가 첫 층을 자동 선택.

handleSave 경로(`fromAccessBlocked` 미전달) 는 기존 4-tier 로직 그대로 유지 —
저장 자동 이동 시엔 접근불가 cp 를 건너뛰는 기존 동작 의도 유지.

## Task 5 — Bug F-a 무한 순회 버그 수정 (260424 현장 피드백)

### 증상 / 사용자 피드백

사용자 증언(2026-04-24):
> "완강기 연구동 3F 에 접근불가 cp A(idx=0), B(idx=1) 만 존재. A에서 확인 →
> B로 이동 ✅. 그런데 B에서 확인을 누르면 **A로 다시 돌아감** → 무한 순환.
> 다음 구역(F-b)으로도 도달 불가능."

### Root Cause

Task 4(`fad7fc5`) 가 도입한 7단계 폴백 체인 중 **단계 4 "현재 층 다른 접근불가 cp
(first)"** 가 범인. B(idx=1) 에서 '확인' 시:

1. 단계 1 (accessible next, i > 1): 없음 (2개 cp 모두 접근불가)
2. 단계 2 (accessible first): 없음
3. 단계 3 (접근불가 next, i > 1): 없음 (cp 가 B 가 마지막)
4. **단계 4 (접근불가 first): A(idx=0) 로 되돌아감** ← 순환의 원인
5. 이후 A 에서 확인 → 단계 3 에서 B 로 이동 → B 에서 단계 4 로 A 로... 무한 순회.

`skipCpId` 가 방금 확인한 cp 하나만 배제하므로, "이전에 dismiss 된 cp" 는 배제 대상이
아니라서 단계 4가 계속 과거로 점프함.

### 수정 — 단계 4 제거 (6단계 체인)

`advanceToNextPending` 의 `fromAccessBlocked=true` 분기에서 **단계 4 (접근불가 first
재탐색) 만 제거**. 나머지는 그대로 유지해 새 6단계 체인으로 재넘버링.

| 기존 단계 | 새 단계 | 설명                                            | 상태         |
| --------- | ------- | ----------------------------------------------- | ------------ |
| 1         | 1       | 현재 층 accessible next cp (i > pickerIdx)      | 유지         |
| 2         | 2       | 현재 층 accessible first cp (재탐색)            | 유지         |
| 3         | 3       | 현재 층 다른 접근불가 cp (next 만, i > pickerIdx) | 유지         |

핵심 설계 결정:

  dismiss 한 cp 로 되돌아가기" 는 자동화 하면 안 된다 (순환 원인).
