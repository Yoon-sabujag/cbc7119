## Design System

| Property | Value |

## Spacing Scale

Declared values (multiples of 4 only). Source: measured from existing pages (`ReportsPage.tsx`, `DashboardPage.tsx`).

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding |
| sm | 8px | Header internal padding, button gaps |
| md | 12px | Default section/card internal padding (matches existing 12px gutter) |
| lg | 16px | Scrollable content area horizontal padding |
| xl | 24px | Inter-section vertical gap |
| 2xl | 32px | Not used — avoid; project uses 12–16px gutters consistently |
| 3xl | 64px | Not used — avoid |

Exceptions:
- Touch targets (buttons, toggle pills): minimum 28px height / 44px touch width on mobile. Existing pattern: `height: 28–34px` for icon buttons.
- Card border-radius: 14px (matches existing `ReportsPage` card `borderRadius: 14`).
- Small badge border-radius: 5–7px (matches existing `StatusBadge` padding pattern).

## Typography

Source: measured from `ReportsPage.tsx`, `DashboardPage.tsx`, `src/components/ui/index.tsx`.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 13px | 700 (bold) | 1.4 | Card titles, schedule item titles, section headings within cards, mode toggle segments, textarea input, download button label |
| Secondary body | 11px | 400 (regular) | 1.4 | Sub-labels, card metadata, empty state body, instruction text, save button label |
| Label / badge | 8–9px | 700 (bold) | 1 | StatusBadge text, DutyChip role label, CatBar-adjacent text |
| Header title | 14px | 700 (bold) | 1.2 | Page header `<header>` title span |

Font weights used: 400 (regular) and 700 (bold). No intermediate weights (500, 600) are introduced by this phase — maintain this constraint.

Monospace exception: percentages and numeric stats use `font-family: JetBrains Mono, monospace` at `font-size: 10px` within the existing `Donut` component. This falls within the Secondary body (11px) size slot as a component-inherited value. The font weight for this element is inherited from the existing `Donut` component — it is not a new weight declaration in this spec.

- The "일별 다운로드" / "월별 다운로드" toggle active state background

**`var(--safe)` reserved for:**
- Completed schedule item row background tint: `rgba(34,197,94,.08)` (lighter than existing `done` StatusBadge tint to avoid visual noise)
- Completed schedule item checkmark icon color: `var(--safe)`
- "완료" StatusBadge (already in existing component — do not modify)

**Second semantic color:** `var(--danger)` for destructive actions only (none expected in Phase 4 — no delete flows).

## Interaction Contract

### Date Navigation

- Default date: today in KST (computed as `new Date()` in browser; user's device is assumed to be in KST).
- For 일별 mode: left/right chevron navigates by ±1 day. Future dates disabled (chevron hidden).
- For 월별 mode: left/right chevron navigates by ±1 month. Current month = max (future months hidden).
- Date change triggers: refetch `GET /api/daily-report?date=YYYY-MM-DD`, regenerate preview, clear `생성 중...` state if download was in progress.

### Mode Toggle

- Toggle between 일별/월별 is instant (no loading). Just changes CTA label and file generation path.
- 특이사항 input persists across mode toggle (bound to selected date, not mode).

### Download Flow

1. User clicks download button.
2. Button enters `생성 중...` state immediately (synchronous UI update).
3. Client assembles `DailyReportData` from cached API response + `shiftCalc` + `dailyReportCalc`.
4. Client runs fflate Excel generation.
5. Browser triggers file download (no navigation, no new tab).
6. Button returns to idle state.
7. On error: toast `엑셀 생성 중 오류가 발생했습니다`, button returns to idle.

### Dashboard Completion Refresh

- Completion state is computed server-side on each `GET /api/dashboard/stats` call.
- React Query `staleTime: 30_000` applies. No manual refresh button needed.
- After a user marks a non-inspect item as manually complete (PATCH), invalidate `['dashboard']` query key so the card re-fetches immediately.
