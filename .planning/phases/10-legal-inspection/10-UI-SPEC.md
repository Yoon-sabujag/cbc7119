## Page Layout Specifications

### LegalPage (`/legal`)

**Primary visual anchor:** The sticky 2-row filter header (status tab strip + year filter) is the focal element. It anchors the user's task context — which inspection round and status they are browsing — and remains visible as the card list scrolls beneath it.

Structure:
1. Self-contained header (48px) — title: "법적 점검"
2. Sticky filter bar:
   - Row 1: status tab strip (`전체` / `미조치` / `완료`) — tab filters by finding status across all rounds
   - Row 2: year filter select (`2024` / `2025` / `2026`) — default current year
3. Scroll area: inspection round cards (schedule_items based)
4. No FAB — findings are created from within a round

**Inspection round card layout** (2 lines):
- Line 1: round title (e.g. `소방 상반기 종합정밀점검`) in `fontSize:14, fontWeight:700, color:var(--t1)` + result badge (right-aligned)
- Line 2: date + agency + finding summary (`지적 N건 · 완료 N건`) in `fontSize:12, color:var(--t2)`
- Left accent border color by result (see Color section)
- Admin-only: tap card shows result edit inline (관리자 전용 "결과 기록" button in card)

### LegalFindingsPage (`/legal/:id`)

Structure:
1. Self-contained header (48px) — title: round subcat label + date (e.g. `종합정밀 2025.06.`)
2. Admin-only sticky sub-header row: result selector (적합/부적합/조건부적합) + PDF upload button
3. Finding list (scrollable) — same card pattern as RemediationPage
4. Fixed bottom CTA: `+ 지적사항 등록` (visible to all roles) — opens BottomSheet

**Finding card layout** (4 lines, matches RemediationPage card):
- Line 1: finding description truncated + status badge (`미조치` / `완료`)
- Line 2: location text (`var(--t2)`, `fontSize:12`)
- Line 3: registered date + registrant name (`var(--t3)`, `fontSize:11`)
- Left accent: danger for `open`, safe for `resolved`

### LegalFindingDetailPage (`/legal/:id/finding/:fid`)

Structure:
1. Self-contained header (48px) — title: "지적 상세"
2. Scrollable content with 3–4 sections (matches RemediationDetailPage):
   - Section "지적 정보": description, location, 등록일, 등록자
   - Section "지적 사진": photo display (if exists) + PhotoButton (all roles)
   - Section "조치 내용 입력" (open state, all roles): textarea + PhotoButton
   - Section "조치 완료" (resolved state): 조치일시, 조치자, 조치 메모, 조치 사진
3. Fixed bottom CTA bar: `조치 완료` button (open state only, all roles)

## Interaction Contract

### Navigation
- All 3 routes (`/legal`, `/legal/:id`, `/legal/:id/finding/:fid`) added to `NO_NAV_PATHS` in `App.tsx`
- Each page renders its own 48px header with back button (`navigate(-1)`)
- SideMenu system section: `soon:true` → `soon:false` for '법적 점검' item

### Tab state
- `/legal` page: status tab (`전체`/`미조치`/`완료`) stored in URL `?tab=` param (matches RemediationPage pattern for back navigation)
- Year filter: component `useState` (not URL param — secondary filter)

### Finding registration BottomSheet
- Opened by `+ 지적사항 등록` button on LegalFindingsPage
- Fields: 지적 내용 (textarea, required), 위치 (text input, optional)
- Backdrop tap closes sheet
- Drag handle visual only (no swipe-to-dismiss implementation required)

### Report PDF viewer
- Open in new tab (`window.open(url, '_blank')`) — per Claude's Discretion (D ref: CONTEXT.md "결과 보고서 PDF 뷰어 방식")
- No in-app PDF viewer required for Phase 10

### Admin result input
- Shown inline on LegalFindingsPage sub-header (admin role only)
- `<select>` for result (적합/부적합/조건부적합), `<button>` for PDF upload
- Same `select` input style as EducationPage: `background: var(--bg3)`, `border: '1px solid var(--bd2)'`

### Scroll clearance
- When fixed CTA bar visible: `paddingBottom: 'calc(72px + var(--sab, 0px))'` on scroll container
- When no CTA bar: `paddingBottom: 24`
