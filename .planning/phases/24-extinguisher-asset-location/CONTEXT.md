# Phase 24 — 소화기 자산-위치 분리

**도메인:** 소방안전 점검, 소화기 자산 관리
**5월 법정점검 준비 작업.** 운영 관찰 모드 예외 — 실전 검증 전 데이터 모델 재설계 필요.

---

## 🎯 목표
도면 마커는 **영구 위치**, 소화기는 위치에 **배치/교체되는 자산**으로 분리. 소화기 교체·이동·폐기를 추적 가능하게 한다.

## 🔥 동기 (Why now)
- 현재 모델은 `check_points` ↔ `extinguishers` 1:1 영구. 소화기 교체 시 동일 위치에서 정보를 갱신할 방법이 없음.
- 분말 소화기 10년 연한 도래·임박 항목들이 실제 교체 대기 상태인데, 교체하려면 폐기 후 새 마커 등록이 필요해 동선이 어색하고 위치 이력이 끊김.
- 5월 법정점검 시 실제 교체가 빈번할 것으로 예상.

---

## 🧱 데이터 모델 변경

### 위치 (불변)
- `floor_plan_markers` + `check_points` 의 위치 필드: `id`, `qr_code`, `floor`, `zone`, `mgmt_no`
- 가변: `location` (개소명) — 마커 수정으로 변경 가능

### 자산 (extinguishers)
- 자산 식별자: 자체 `id`
- 속성: `type`, `prefix_code`, `seal_no`, `serial_no`, `approval_no`, `manufactured_at`, `manufacturer`
- **신규 컬럼**: `check_point_id` (nullable, 현재 매핑 위치) — 이미 있을 가능성 있음, 확인 필요
- **신규 컬럼**: `status` ('active' / '폐기', default 'active')

### 점검 기록 (check_records)
- **신규 컬럼**: `extinguisher_id` (nullable, 점검 시점 소화기 스냅샷)

### 마이그레이션
- 기존 1:1 매핑은 그대로 보존 (extinguishers.check_point_id 그대로 유지)
- 기존 행 모두 status='active'
- 기존 check_records.extinguisher_id 는 가능한 경우 추론해서 채움 (현재 매핑된 ext) — 또는 NULL

## 🆕 신규 페이지: 소화기 리스트

라우트: `/extinguishers` (또는 적절한 경로)

### 목록 화면
- 필터: 매핑 상태(전체 / 미배치 / 매핑 / 폐기) + zone + floor + type + 검색어
- 헤더에 「+ 새로 등록」 버튼

### 신규 등록 폼
- 입력: 종류·접두문자·증지번호·제조번호·형식승인·제조년월·제조업체
- 안내 배너: *"등록 후 한 번에 최대 3개 필드만 수정 가능합니다."*
- 「저장」 → 미배치 상태로 등록
- (마커 동행 시) 「저장」 → 동행 마커에 **자동 매핑**

### 카드 상세 모드 (카드 클릭으로 토글)
| 상태 | 버튼 |
|---|---|
| 미매핑 + 미점검 (방금 등록) | 「정보 수정」 · 「소화기 배치」 · **「삭제」** (hard delete) |
| 미매핑 + 점검 기록 있음 | 「정보 수정」 · 「소화기 배치」 · **「폐기」** (soft, status='폐기') |
| 매핑됨 | 「정보 수정」 · **「소화기 분리」** |
| 폐기됨 | 조회만 |

### 정보 수정 모달
- 모든 필드 수정 가능
- **한 번에 최대 3개 필드 변경** — 4개 이상이면 카운터 빨강 + 저장 비활성 + 안내 *"4개 이상 변경하려면 폐기+재등록을 사용하세요"*
- 변경 수 실시간 표시 (예: `변경: 2 / 3`)
- 백엔드도 동일 검증

## 🗺️ 도면 페이지 변경 (소화기/소화전 plan type)

### 마커 추가 모달 (편집 모드)
- 입력: **개소명** (구 라벨칸) + **구역** (필수)
- 층은 도면 페이지에서 이미 선택된 층 자동 사용 (입력 X)
- 매핑은 옵션: 마커 생성만으로 충분 (빈 마커 ❓), 매핑은 「소화기 배치」 흐름으로 별도
- ❌ 제거: 소화기 종류 선택 / 기존 개소 연결

### 마커 수정 모달 (편집 모드)
- 개소명 변경
- 「**소화기 배치**」 → 리스트 페이지로 이동 (마커 동행)
- 「**소화기 분리**」 → 미배치 상태로 (폐기 X), 마커는 ❓ 로 전환
- ❌ 제거: 소화기 종류 / 점검 개소 연결

### 마커 클릭 (점검 모드)
- 매핑된 마커 → 기존 점검 흐름 그대로
- ❓ 마커 → "소화기 미배치" 안내 + 「소화기 배치하기」 버튼 (리스트로 이동)
- 점검 기록 입력 모달의 소화기 정보 카드에는 「정보 수정」(≤3) · 「소화기 분리」 버튼 추가

### 빈 마커 (미배치) 표시
- 색: 빨강 (`#ef4444` / `var(--danger)`)
- 모양: ❓ (물음표)
- **범례 status row 에 「미배치」 항목 추가** (status row 일관성 유지)

## 🔄 통합 동선 (양방향)

### 도면 마커 수정 → 「소화기 배치」 → 리스트 페이지 (마커 동행)
| 사용자 행동 | 결과 |
|---|---|
| 매핑된 다른 소화기 클릭 | "X 위치 소화기. 서로 바꿀까요?" → **위치 스왑** (양쪽 동시 변경) |
| 미배치 소화기 클릭 | 동행 마커에 매핑 |
| 「+ 새로 등록」 후 저장 | 동행 마커에 **자동 매핑** |

### 리스트 미배치 카드 → 「소화기 배치」 → 도면 페이지
- 도면 페이지로 이동 (소화기 ID + 소화기의 zone 동행)
- 소화기에 zone/floor 정보가 있으면 자동 layer 이동, 없으면 사용자가 층 선택
- 빈 ❓ 마커 클릭 → "여기에 배치할까요?" 확인 → 매핑
- 리스트로 자동 복귀

## 🔌 신규 API 엔드포인트

| 엔드포인트 | 메소드 | 설명 |
|---|---|---|
| `/api/extinguishers/:id` | PUT | 정보 수정 (변경 필드 ≤ 3 검증) |
| `/api/extinguishers/:id/assign` | POST | 위치 매핑 (body: `{ check_point_id }`) |
| `/api/extinguishers/:id/unassign` | POST | 위치 분리 (status는 active 유지) |
| `/api/extinguishers/:id/swap` | POST | 위치 스왑 (body: `{ other_extinguisher_id }`) |
| `/api/extinguishers/:id/dispose` | POST | 폐기 처리 (status='폐기' + check_point_id=NULL) |
| `/api/extinguishers/:id` | DELETE | hard delete (미매핑+미점검 항목만) |

기존 유지: `GET /api/extinguishers/:checkPointId`, `GET /api/extinguishers`, `POST /api/extinguishers/create`

## ✅ 성공 기준

1. 소화기 리스트 페이지에서 신규 등록·수정(≤3필드)·삭제·폐기·분리가 가능하다
2. 도면 마커 편집에서 위치 추가·개소명 수정·매핑 변경(스왑/배치/분리) 가능하다
3. 소화기 교체 시 기존은 폐기로 보존되고 새 자산이 동일 위치에 매핑되어 이력 추적된다
4. 점검 기록(check_records)에 그 시점 소화기 ID가 스냅샷으로 저장된다
5. 빈 마커는 빨간 ❓ 로 도면+범례에 표시되고, 점검 모드에서 ❓ 클릭 시 매핑 유도 안내가 뜬다
6. 리스트 ↔ 도면 양방향 매핑 동선이 마커 컨텍스트 동행으로 끊김 없이 동작한다
7. 기존 1:1 데이터가 손실 없이 새 모델로 보존된다 (마이그레이션 검증)

## ⚠️ 위험·고려사항

- **데이터 마이그레이션**: 기존 운영 데이터 손실 없이 컬럼 추가만 (DDL)
- **점검 기록 무결성**: extinguisher_id 추가 후 기존 행 NULL — 새 점검부터 채워짐, 과거 기록은 NULL 으로 유지 (또는 현재 매핑된 ext_id 추정해서 채움)
- **PWA 캐시**: 배포 후 "여전히 똑같다" 클레임 발생 가능 — 사용자에게 앱 재설치/새로고침 안내
- **법정점검 일정 임박**: 5월 첫 주가 마감. 작업은 빠르게 진행하되 회귀 발생 안 하도록 신중히
- **호환성**: 기존 `extinguishers/create` 엔드포인트는 그대로 유지 (FloorPlanPage 의 마커-편집 새로 등록 흐름이 사용 중) — 다만 이번 phase 에서 그 흐름은 사라지고 리스트 페이지 흐름으로 대체되므로 deprecate 가능. 리팩토링 시점은 plan 단계에서 결정.
