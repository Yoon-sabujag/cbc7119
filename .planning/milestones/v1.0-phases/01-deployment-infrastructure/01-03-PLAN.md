<!-- Commands needed for D1 migration verification. From RESEARCH.md Pattern 4. -->

D1 migration commands (run from cha-bio-safety/ directory):
```bash
# Compare state — always pass --local or --remote explicitly (never rely on default)
npx wrangler d1 migrations list cha-bio-db --local
npx wrangler d1 migrations list cha-bio-db --remote

# Apply unapplied migrations to remote (skips already-applied, rolls back on failure)
npx wrangler d1 migrations apply cha-bio-db --remote --yes

# Check if d1_migrations tracking table exists in remote
npx wrangler d1 execute cha-bio-db --remote --command "SELECT name FROM sqlite_master WHERE type='table' AND name='d1_migrations';"
```

CRITICAL: The deploy.sh script uses `wrangler d1 execute --file` (no tracking table, re-applies everything).
The CI workflow uses `wrangler d1 migrations apply --remote --yes` (tracking table, skips applied).
If deploy.sh was the only migration tool used before this phase, the d1_migrations tracking table may NOT exist in the remote DB, meaning `migrations apply` will attempt to re-apply all 23 migrations.

Open Question 2 from RESEARCH.md: Before running `migrations apply`, check if the tracking table exists. If it does not exist, the command will try to run all 23 migrations against a DB that already has all the schema — this will cause errors like "table already exists". In that case:
- Do NOT panic — wrangler handles this by wrapping in transactions
- If migrations fail with "table already exists", the tracking table still gets created with those migrations marked as applied
- As a safety check: run `npx wrangler d1 execute cha-bio-db --remote --command ".schema"` and compare with local migrations to confirm the full schema is present

Production URL: https://cbc7119.pages.dev
